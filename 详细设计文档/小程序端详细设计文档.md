## 1. 技术选型与约束

### 1.1 技术栈

| 类别         | 选型                              | 说明                                         |
| ------------ | --------------------------------- | -------------------------------------------- |
| 框架         | **Taro 3.x**                     | 跨端编译框架，编译为微信/支付宝小程序        |
| 开发语言     | **TypeScript**                    | 严格类型检查，提升代码质量                   |
| 语法         | **React 18** (Hooks)              | 函数式组件 + Hooks，不使用 Class Component   |
| UI 组件库    | **NutUI React**                   | 京东 Taro 官方推荐组件库，深度适配小程序     |
| 状态管理     | **zustand**                       | 轻量级状态管理，TypeScript 友好              |
| 网络请求     | **Taro.request** + 自封装        | 基于 Taro 原生 API 封装统一请求层            |
| 样式方案     | **SCSS** + BEM 命名              | 全局变量 + 模块化样式                        |
| 代码规范     | **ESLint** + **Prettier**        | 统一代码风格                                 |
| 构建工具     | Taro CLI 内置 (Webpack 5)        | 默认构建配置                                 |

### 1.2 编译目标

| 平台          | 优先级 | 说明                                   |
| ------------- | ------ | -------------------------------------- |
| 微信小程序    | P0     | 主要目标平台，首发                     |
| 支付宝小程序  | P1     | 兼容适配，二期发布                     |

### 1.3 设计约束

- **主包体积** ≤ 2MB，单个分包 ≤ 2MB，总包 ≤ 20MB（微信小程序限制）
- **首屏加载** < 1s（分包 + 预加载 + 骨架屏）
- **接口响应** < 300ms（依赖后端 SLA）
- 所有页面支持**下拉刷新**和**触底加载**
- 网络异常时展示**离线占位页**，非空白崩溃

## 4. 全局状态管理设计

### 4.1 Store 划分

采用 zustand 按业务域拆分 Store，各 Store 独立管理状态与操作方法。

#### 4.1.1 useUserStore — 用户与登录态

```typescript
interface UserState {
  // 状态
  token: string | null;
  refreshToken: string | null;
  userInfo: UserInfo | null;
  currentRole: 'receiver' | 'runner';  // 当前激活角色
  isLoggedIn: boolean;

  // 操作
  login: (phone: string, code: string) => Promise<void>;
  logout: () => void;
  refreshToken: () => Promise<void>;
  fetchUserInfo: () => Promise<void>;
  switchRole: (role: 'receiver' | 'runner') => void;
  updateProfile: (data: Partial<UserInfo>) => Promise<void>;
}

interface UserInfo {
  id: number;
  phone: string;         // 脱敏后
  nickname: string;
  avatar: string;
  user_type: number;     // 1-收件人, 2-跑腿员
  runner_status: number; // 0-未申请, 1-审核中, 2-已通过, 3-已拒绝
  credit_score: number;
  is_blacklisted: boolean;
}
```

## 5. 网络层封装

### 5.1 请求封装（api/request.ts）

统一封装 `Taro.request`，实现以下能力：

| 能力           | 实现方式                                                     |
| -------------- | ------------------------------------------------------------ |
| 基础 URL       | 根据环境变量配置 `BASE_URL`（dev / prod）                   |
| Token 注入     | 请求拦截器自动从 `useUserStore` 读取 Token 并添加 `Authorization` 头 |
| 统一响应解析   | 响应拦截器解析 `{code, msg, data, trace_id}` 结构           |
| 业务错误处理   | `code !== 0` 时抛出 `BusinessError`，携带 code + msg        |
| Token 自动刷新 | 收到 `10002` (Token 失效) 时自动调用 `/auth/refresh`，成功后重放原请求 |
| 请求重试       | 网络超时自动重试 1 次（幂等请求）                            |
| 全局 Loading   | 可选配置是否展示 loading 态                                  |
| 请求头注入     | 自动注入 `X-Client-Type: miniapp`、`X-Client-Version`、地理位置（可选） |

### 5.2 请求层伪代码

```typescript
// api/request.ts
import Taro from '@tarojs/taro';
import { useUserStore } from '@/stores/useUserStore';

const BASE_URL = process.env.TARO_APP_API_BASE;

interface ApiResponse<T> {
  code: number;
  msg: string;
  data: T;
  trace_id: string;
}

interface RequestConfig {
  url: string;
  method?: 'GET' | 'POST' | 'PUT' | 'DELETE';
  data?: any;
  params?: Record<string, any>;
  headers?: Record<string, string>;
  showLoading?: boolean;
  withGeo?: boolean;        // 是否注入地理位置
  retry?: boolean;          // 是否允许重试
}

export async function request<T>(config: RequestConfig): Promise<T> {
  const { token } = useUserStore.getState();

  // 构建请求头
  const headers: Record<string, string> = {
    'Content-Type': 'application/json',
    'X-Client-Type': 'miniapp',
    'X-Client-Version': getAppVersion(),
    ...config.headers,
  };
  if (token) {
    headers['Authorization'] = `Bearer ${token}`;
  }

  // 地理位置注入
  if (config.withGeo) {
    try {
      const loc = await Taro.getLocation({ type: 'gcj02' });
      headers['X-Geo-Lat'] = String(loc.latitude);
      headers['X-Geo-Lng'] = String(loc.longitude);
    } catch { /* 用户拒绝授权地理位置，后端会做风控 */ }
  }

  // 发起请求
  const response = await Taro.request({
    url: `${BASE_URL}${config.url}`,
    method: config.method || 'GET',
    data: config.data,
    header: headers,
  });

  const body = response.data as ApiResponse<T>;

  // 业务错误处理
  if (body.code !== 0) {
    // Token 失效 → 自动刷新
    if (body.code === 10002) {
      const refreshed = await tryRefreshToken();
      if (refreshed) return request<T>(config); // 重放
    }
    throw new BusinessError(body.code, body.msg);
  }

  return body.data;
}
```

### 5.3 API 模块示例

```typescript
// api/parcel.ts
import { request } from './request';
import type { Parcel, PickupCodeInfo, PaginatedList } from './types';

export const parcelApi = {
  /** 我的包裹列表 */
  getMy: (params: { status?: number; page?: number; page_size?: number }) =>
    request<PaginatedList<Parcel>>({
      url: '/parcels/my',
      params,
    }),

  /** 包裹详情 */
  getDetail: (id: number) =>
    request<Parcel>({
      url: `/parcels/${id}`,
    }),

  /** 获取取件码（含二维码） */
  getPickupCode: (id: number) =>
    request<PickupCodeInfo>({
      url: `/parcels/${id}/pickup-code`,
    }),

  /** 自助出库 */
  selfCheckout: (data: { pickup_code: string; station_id: number }) =>
    request<{ parcel_id: number; pickup_time: string }>({
      url: '/pickup/self-checkout',
      method: 'POST',
      data,
      withGeo: true,  // 自动注入地理位置
    }),

  /** 扫驿站码批量出库 */
  scanStation: (data: { station_qr: string; pickup_codes: string[] }) =>
    request<{ success: any[]; failed: any[] }>({
      url: '/pickup/scan-station',
      method: 'POST',
      data,
      withGeo: true,
    }),
};
```

## 8. 关键技术实现

### 8.1 登录态管理与自动刷新

```
用户打开小程序
  │
  ▼
检查本地 Token
  │
  ├── 无 Token → 跳转登录页
  │
  └── 有 Token → 尝试调用 /user/info
       │
       ├── 成功 → 进入首页
       │
       └── 返回 10002 (Token 失效)
            │
            ├── 有 refreshToken → 调用 /auth/refresh
            │    │
            │    ├── 成功 → 更新 Token → 重试原请求
            │    │
            │    └── 失效 → 清除本地 → 跳转登录页
            │
            └── 无 refreshToken → 跳转登录页
```

### 8.2 地理位置风控

```typescript
// hooks/useGeoLocation.ts
export function useGeoLocation() {
  const [location, setLocation] = useState<{ lat: number; lng: number } | null>(null);
  const [authStatus, setAuthStatus] = useState<'authorized' | 'denied' | 'undetermined'>('undetermined');

  const requestLocation = async () => {
    try {
      const res = await Taro.getLocation({ type: 'gcj02' });
      setLocation({ lat: res.latitude, lng: res.longitude });
      setAuthStatus('authorized');
    } catch (err) {
      setAuthStatus('denied');
    }
  };

  // 首次使用引导授权
  const ensureLocation = async () => {
    const setting = await Taro.getSetting();
    if (setting.authSetting['scope.userLocation'] === false) {
      // 用户曾拒绝，引导打开设置
      showLocationGuide();
      return null;
    }
    if (!location) await requestLocation();
    return location;
  };

  return { location, authStatus, ensureLocation };
}
```

**降级策略**：
- 用户拒绝地理位置授权 → 允许使用小程序，但自助出库功能不可用（提示需要授权）
- 取件码展示不受影响，仅影响自助核销流程

### 8.5 下拉刷新与分页加载

```typescript
// hooks/usePagination.ts
export function usePagination<T>(
  fetchFn: (params: { page: number; page_size: number }) => Promise<PaginatedList<T>>,
  pageSize = 20,
) {
  const [list, setList] = useState<T[]>([]);
  const [page, setPage] = useState(1);
  const [hasMore, setHasMore] = useState(true);
  const [loading, setLoading] = useState(false);

  const refresh = async () => {
    setLoading(true);
    const res = await fetchFn({ page: 1, page_size: pageSize });
    setList(res.list);
    setPage(1);
    setHasMore(res.list.length >= pageSize);
    setLoading(false);
  };

  const loadMore = async () => {
    if (!hasMore || loading) return;
    setLoading(true);
    const nextPage = page + 1;
    const res = await fetchFn({ page: nextPage, page_size: pageSize });
    setList(prev => [...prev, ...res.list]);
    setPage(nextPage);
    setHasMore(res.list.length >= pageSize);
    setLoading(false);
  };

  return { list, loading, hasMore, refresh, loadMore };
}
```

页面配置 `enablePullDownRefresh: true`，在 `onPullDownRefresh` 中调用 `refresh()`，在 `onReachBottom` 中调用 `loadMore()`。

## 10. 性能优化策略

### 10.1 加载优化

| 策略             | 实现方式                                             |
| ---------------- | ---------------------------------------------------- |
| 分包加载         | 按业务域拆分分包，主包仅保留首页、登录、个人中心     |
| 预加载           | 配置 preloadRule，在空闲时预加载邻近分包             |
| 骨架屏           | 首页、列表页使用骨架屏占位，提升感知加载速度         |
| 图片懒加载       | 列表中图片使用 `lazy-load` 属性                      |
| 图片压缩         | 静态资源使用 WebP 格式，CDN 加速                     |

### 10.2 渲染优化

| 策略             | 实现方式                                             |
| ---------------- | ---------------------------------------------------- |
| 虚拟列表         | 包裹列表、任务大厅列表超过 50 条时使用虚拟列表       |
| 减少 setData     | 使用 zustand 局部订阅，避免不必要的组件重渲染        |
| 防抖/节流        | 搜索输入防抖 300ms，滚动触底节流 500ms               |
| 图片占位         | 图片组件设置固定宽高比，避免布局抖动                 |

### 10.3 网络优化

| 策略             | 实现方式                                             |
| ---------------- | ---------------------------------------------------- |
| 请求缓存         | 快递公司字典等低频变更数据缓存到本地，定时更新       |
| 接口合并         | 首页进入时并行请求包裹列表 + 未读数 + 用户信息       |
| 弱网适配         | 请求超时设置 10s，超时后展示重试按钮                  |
| 离线缓存         | 最近一次包裹列表缓存到本地，离线时展示缓存数据       |
