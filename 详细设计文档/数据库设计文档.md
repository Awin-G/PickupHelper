## 1. 设计原则

| 原则 | 说明 |
| ---- | ---- |
| **无 ORM** | 使用 sqlx + 原生 SQL，无 ORM 抽象层，SQL 与 Go 代码一一对应 |
| **参数化查询** | 全部使用 `?` 占位符，禁止字符串拼接，防 SQL 注入 |
| **单表职责** | 每个表只负责一类实体，避免过度关联 |
| **索引驱动** | 所有查询路径均有对应索引支持，避免全表扫描 |
| **乐观锁优先** | 货架容量等并发场景使用 `version` 乐观锁 |
| **事务明确** | 跨表操作必须显式声明事务边界 |
| **幂等设计** | 写操作支持 `Idempotency-Key`，避免重复提交 |
| **状态机约束** | 状态变更通过 SQL 条件判断实现，禁止应用层直接改状态 |

## 2. 表结构总览

| 表名 | 用途 | 核心约束 |
| ---- | ---- | -------- |
| `stations` | 驿站基础信息 | `name` 唯一 |
| `users` | 用户注册信息 | `phone`/`openid` 唯一 |
| `admins` | 管理员账号 | `username` 唯一 |
| `runner_applications` | 跑腿员资质申请 | `user_id` 外键，`status` 状态机 |
| `parcels` | 包裹主数据 | `tracking_no+station_id` 唯一，`pickup_code+station_id` 唯一 |
| `pickup_logs` | 取件操作日志 | `parcel_id` 外键 |
| `proxy_orders` | 代取订单 | `parcel_id` 唯一（一对一） |
| `shelf_layout` | 货架布局与容量 | `station_id+shelf_code` 唯一，`version` 乐观锁 |
| `notifications` | 消息通知记录 | `user_id` 外键，`send_status` 状态 |
| `courier_companies` | 快递公司字典 | `company_name`/`company_code` 唯一 |
| `operation_logs` | 系统操作审计 | 异步写入，非事务 |

## 4. 各表 CRUD SQL 设计

### 4.1 驿站信息表 stations

#### 4.1.1 插入（新增驿站）

```sql
INSERT INTO `stations` (`name`, `address`, `latitude`, `longitude`, `business_hours`, `status`)
VALUES (?, ?, ?, ?, ?, ?);
```

| 参数 | 类型 | 必填 | 说明 |
| ---- | ---- | ---- | ---- |
| name | varchar(100) | 是 | 驿站名称 |
| address | varchar(255) | 否 | 详细地址 |
| latitude | decimal(10,7) | 否 | 纬度 |
| longitude | decimal(10,7) | 否 | 经度 |
| business_hours | varchar(100) | 否 | 默认 `09:00-20:00` |
| status | tinyint | 否 | 默认 1 |

**使用场景**：系统管理员新增驿站

**异常处理**：
- `uk_name` 冲突 → 返回错误码 10701（名称已存在）
- 经纬度格式错误 → 应用层校验拦截

---

#### 4.1.2 查询列表

```sql
SELECT `id`, `name`, `address`, `latitude`, `longitude`, `business_hours`, `status`, `created_at`
FROM `stations`
WHERE (`name` LIKE CONCAT('%', ?, '%') OR `address` LIKE CONCAT('%', ?, '%'))
  AND `status` = ?
ORDER BY `created_at` DESC
LIMIT ?, ?;
```

| 参数 | 类型 | 必填 | 说明 |
| ---- | ---- | ---- | ---- |
| keyword | string | 否 | 名称/地址模糊搜索 |
| keyword | string | 否 | 同前（复用） |
| status | tinyint | 否 | 状态筛选，不传则不限 |
| offset | int | 是 | 分页偏移 |
| limit | int | 是 | 每页条数 |

**使用场景**：系统管理员查看驿站列表

---

#### 4.1.3 查询详情

```sql
SELECT * FROM `stations` WHERE `id` = ?;
```

**使用场景**：查看单个驿站详情

**异常处理**：记录不存在 → 返回 404

---

#### 4.1.4 更新

```sql
UPDATE `stations`
SET `name` = ?, `address` = ?, `latitude` = ?, `longitude` = ?, `business_hours` = ?, `status` = ?
WHERE `id` = ?;
```

**使用场景**：编辑驿站信息

**事务要求**：无跨表操作，单语句即可

---

#### 4.1.5 删除

```sql
DELETE FROM `stations` WHERE `id` = ?;
```

**使用场景**：删除驿站

**前置校验**：需检查该驿站下是否存在包裹/订单，若有则拒绝删除

### 4.2 用户表 users

#### 4.2.1 插入（注册）

```sql
INSERT INTO `users` (`phone`, `nickname`, `avatar`, `openid`, `user_type`)
VALUES (?, ?, ?, ?, ?)
ON DUPLICATE KEY UPDATE `updated_at` = NOW();
```

| 参数 | 类型 | 必填 | 说明 |
| ---- | ---- | ---- | ---- |
| phone | varchar(11) | 是 | 手机号 |
| nickname | varchar(50) | 否 | 默认空 |
| avatar | varchar(255) | 否 | 默认空 |
| openid | varchar(64) | 否 | 微信 openid |
| user_type | tinyint | 否 | 默认 1（普通用户） |

**使用场景**：手机号验证码登录时自动注册

**设计说明**：`ON DUPLICATE KEY UPDATE` 实现幂等，同手机号重复注册不报错

---

#### 4.2.2 查询（按手机号）

```sql
SELECT * FROM `users` WHERE `phone` = ? LIMIT 1;
```

**使用场景**：验证码登录校验

---

#### 4.2.3 查询（按 openid）

```sql
SELECT * FROM `users` WHERE `openid` = ? LIMIT 1;
```

**使用场景**：微信登录校验

---

#### 4.2.4 更新（个人信息）

```sql
UPDATE `users` SET `nickname` = ?, `avatar` = ? WHERE `id` = ?;
```

**使用场景**：用户修改昵称/头像

---

#### 4.2.5 更新（升级为跑腿员）

```sql
UPDATE `users` SET `user_type` = 2, `runner_status` = 2 WHERE `id` = ? AND `user_type` = 1;
```

**使用场景**：跑腿员申请审核通过后升级身份

**设计说明**：`AND user_type = 1` 条件确保只能从普通用户升级，防止重复操作

---

#### 4.2.6 更新（黑名单）

```sql
UPDATE `users` SET `is_blacklisted` = ?, `updated_at` = NOW() WHERE `id` = ?;
```

**使用场景**：管理员拉黑/解除拉黑用户

### 4.4 跑腿员申请表 runner_applications

#### 4.4.1 插入

```sql
INSERT INTO `runner_applications` (`user_id`, `real_name`, `student_id`, `id_card_image`, `status`)
VALUES (?, ?, ?, ?, 1);
```

**使用场景**：用户提交跑腿员申请

**前置校验**：检查用户是否已有申请（`WHERE user_id = ?`），若有则拒绝

#### 4.4.2 查询列表

```sql
SELECT a.*, u.phone, u.nickname
FROM `runner_applications` a
LEFT JOIN `users` u ON a.user_id = u.id
WHERE a.status = ?
ORDER BY a.created_at DESC
LIMIT ?, ?;
```

**使用场景**：管理员查看申请列表

---

#### 4.4.3 更新（审核）

```sql
UPDATE `runner_applications`
SET `status` = ?, `audit_admin_id` = ?, `audit_remark` = ?
WHERE `id` = ? AND `status` = 1;
```

| 参数 | 说明 |

## 5. 核心业务 SQL 矩阵

### 5.1 包裹入库流程

| 步骤 | SQL | 表 | 事务要求 |
| ---- | --- | --- | -------- |
| 1 | 查询可用货架 | SELECT FROM shelf_layout | 否 |
| 2 | 乐观锁更新货架容量 | UPDATE shelf_layout | 是（与步骤 3 同一事务） |
| 3 | 插入包裹记录 | INSERT INTO parcels | 是 |
| 4 | 创建入库通知 | INSERT INTO notifications | 否（异步） |
| 5 | 记录操作日志 | INSERT INTO operation_logs | 否（异步） |

**事务边界**：步骤 2 和 3 必须在同一事务中

**异常处理**：
- 步骤 2 乐观锁失败 → 重试（最多 3 次），重试失败则回滚并返回错误
- 步骤 3 取件码冲突 → 回滚并重新生成取件码后重试

**事务边界**：步骤 2、3、4 必须在同一事务中

**异常处理**：
- 步骤 1 包裹不存在或状态非「待取」→ 返回错误

## 6. 事务边界与并发控制

### 6.1 事务边界规范

| 场景 | 事务范围 | 涉及表 |
| ---- | -------- | ------ |
| 包裹入库 | 货架更新 + 包裹插入 | shelf_layout, parcels |
| 取件核销 | 包裹状态 + 货架容量 + 取件日志 | parcels, shelf_layout, pickup_logs |
| 跑腿员审核通过 | 申请表状态 + 用户身份 | runner_applications, users |
| 代取接单 | 订单状态更新 | proxy_orders |
| 代取送达确认 | 订单状态更新 | proxy_orders |

### 6.2 乐观锁机制

货架容量更新采用乐观锁：

```sql
-- 增加容量（入库）
UPDATE `shelf_layout`
SET `current_capacity` = `current_capacity` + 1, `version` = `version` + 1
WHERE `id` = ? AND `version` = ?;

-- 减少容量（出库）
UPDATE `shelf_layout`
SET `current_capacity` = `current_capacity` - 1, `version` = `version` + 1
WHERE `id` = ? AND `version` = ? AND `current_capacity` > 0;
```

**重试策略**：
- 最多重试 3 次
- 每次重试前重新查询最新 `version`
- 3 次失败则返回错误，提示管理员检查货架状态

### 6.3 取件码唯一性保障

取件码在驿站内全局唯一（`uk_pickup_code_station`），生成流程：

```
1. 生成随机 6 位数字
2. Redis 检查是否存在（SETNX，TTL 5min）
3. 数据库 INSERT（唯一约束兜底）
4. 若 INSERT 失败（唯一键冲突），回到步骤 1 重试（最多 10 次）
```

### 6.4 分布式锁（定时任务）

定时任务使用 Redis 分布式锁确保单实例执行：

```sql
-- 获取锁（1800秒过期）
SET lock:cron:name unique_value NX EX 1800

-- 释放锁（校验 value 防止误删）
EVAL "if redis.call('get', KEYS[1]) == ARGV[1] then return redis.call('del', KEYS[1]) else return 0 end" 1 lock:cron:name unique_value
```

## 7. 索引策略与性能优化

### 7.1 索引总览

| 表 | 索引 | 用途 |
| --- | --- | --- |

| users | uk_phone, uk_openid | 登录查询 |
| users | idx_user_type, idx_runner_status | 用户类型筛选 |
| admins | uk_username | 登录查询 |
| parcels | uk_tracking_station | 防止重复入库 |
| parcels | uk_pickup_code_station | 取件码唯一性 |
| parcels | idx_receiver_phone, idx_receiver_user_id | 我的包裹列表 |
| parcels | idx_station_status, idx_station_storage_time | 管理员筛选、定时任务扫描 |
| proxy_orders | uk_parcel | 一对一约束 |
| proxy_orders | idx_created_at | 超时扫描 |

| notifications | idx_user_read | 用户通知列表 |
| operation_logs | idx_module_action, idx_created | 审计查询 |

### 7.2 查询性能建议

1. **避免 SELECT ***：只查询需要的字段，减少数据传输
2. **分页优化**：使用 `LIMIT offset, limit`，offset 过大时改用游标分页
3. **复合索引顺序**：等值条件在前，范围条件在后（如 `(station_id, status)` 优于 `(status, station_id)`）
4. **覆盖索引**：对于高频查询，考虑创建覆盖索引避免回表
5. **定时任务批量处理**：使用 `LIMIT` 分批处理，避免单次扫描过多数据

### 7.3 写性能建议

1. **批量插入**：使用 `Prepare` + 循环 `Exec`，在同一事务中执行
2. **异步日志**：操作日志通过缓冲通道异步写入，不阻塞主流程
3. **连接池配置**：根据预期并发量调整 `max_open_conns` 和 `max_idle_conns`
4. **避免长事务**：事务中避免网络 IO（如调用外部 API）

## 8. 数据迁移与版本管理

### 8.1 迁移工具

使用 `golang-migrate` 进行数据库迁移管理。

### 8.2 迁移文件命名规范

```
{version}_{description}.up.sql
{version}_{description}.down.sql
```

### 8.3 初始迁移示例

```
000001_init_schema.up.sql
000001_init_schema.down.sql
000002_add_indexes.up.sql
000002_add_indexes.down.sql
```

### 8.4 迁移执行命令

```bash
# 执行所有待执行迁移
migrate -path ./migrations -database "mysql://user:password@tcp(host:port)/dbname" up

# 回滚最近一次迁移
migrate -path ./migrations -database "mysql://user:password@tcp(host:port)/dbname" down 1

# 查看当前版本
migrate -path ./migrations -database "mysql://user:password@tcp(host:port)/dbname" version
```
